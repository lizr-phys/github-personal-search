Precision@10 不与 18 查询版本直接比较：新增的多数领域每个查询只标注了一个高相关种子，因此其理论 Precision@10 上限为 0.1；Recall、NDCG 与 MRR 更适合衡量这轮领域扩展。

簇覆盖下降是有意的精度/多样性再平衡：旧实现用弱相关项目填满 10 个，造成簇数虚高；新实现拒绝低证据填充，同时所有候选充足的前 10 仍满足至少 3 簇。

## 实施内容

1. 查询规范化加入中英文词典、技术实体、缩写、常见拼写错误、Topic 映射、置信度和缓存。
2. 词法、Topic、画像、趋势、关系和探索先低成本召回，通过 Reciprocal Rank Fusion 合并；语义只处理有界候选。
3. 语言、项目类型、部署、平台、难度、成熟度、许可证、排除项和用户屏蔽在排序前后双重门控。
4. 语义/词法/结构化分数重新校准，增加最低证据阈值、召回来源可信度、归档/Fork/镜像/陈旧/重复惩罚。
5. 精确、综合、灵感、最新分别使用不同探索与时效偏置；最终用配额 + MMR 控制组织、语言、框架和用途重复。
6. 排除语言采用规范化后的精确语言等值判断；`不要 Java` 不再把 JavaScript/TypeScript 当作子串过滤，正向语言数组也会移除已排除语言。

## 推荐验证

- 暖队列最多 50 项，首批固定 10 项；单元测试验证唯一性和配额。
- 30 日曝光抑制继续生效；“已看过”采用 7 日半衰期，显式负反馈采用 30 日半衰期。
- 对已预取的未来批次给出反馈时，批次及尚未发生的曝光会失效，下一批重新按会话画像排序。
- 行为权重全部由配置读取，`已用于项目 +8`、`已运行/复现 +6`、`学习 +5`、`收藏 +4`、`打开 +2`、`不感兴趣 -6` 等均有确定性测试。

## 局限

60 项人工种子目录和 27 条查询只能验证回归方向，不能代表全 GitHub 的绝对 Recall。当前没有足量真实点击标签来训练 LTR；后续应扩大盲标集并按用户群体分层评测，之后再决定是否引入 LambdaMART 或 contextual bandit。
